# Error Handling and Logging Standard

## Overview

All API error responses follow a single machine-readable shape derived from gRPC status
codes mapped to HTTP. Every response includes a correlation ID so clients and operators
can correlate API errors with server logs without exposing internal stack traces.

## Error Response Shape

```json
{
  "error": "NOT_FOUND",
  "message": "Extension not found: ext_abc123",
  "request_id": "req-c8f2a1b0-4e3d"
}
```

| Field        | Type   | Description                                              |
|--------------|--------|----------------------------------------------------------|
| `error`      | string | Machine-readable RPC error code (see table below)        |
| `message`    | string | Human-readable description safe to surface to operators  |
| `request_id` | string | Echoed from `X-Request-ID` response header               |

The `request_id` value is generated by Fastify (`req.id`) and emitted as
`X-Request-ID` on every response so log aggregators and clients can join on it.

## RPC Error Codes

| Code                | HTTP | Meaning                                                  |
|---------------------|------|----------------------------------------------------------|
| `NOT_FOUND`         | 404  | Resource does not exist or is not visible to this tenant |
| `INVALID_ARGUMENT`  | 400  | Client sent malformed input or a constraint was violated |
| `UNAUTHENTICATED`   | 401  | Missing or invalid credential                            |
| `PERMISSION_DENIED` | 403  | Credential is valid but lacks the required capability    |
| `ALREADY_EXISTS`    | 409  | Resource with the same identity already exists           |
| `RESOURCE_EXHAUSTED`| 429  | Rate limit or quota exceeded                             |
| `INTERNAL`          | 500  | Unexpected server-side fault                             |
| `UNAVAILABLE`       | 503  | Upstream dependency (DB, FreeSWITCH) is unreachable      |

No other HTTP error codes are introduced. 422 is mapped to `INVALID_ARGUMENT`.

## Implementation Contract

### Server

- Fastify's global `setErrorHandler` catches every unhandled throw and Fastify
  validation errors, converting them to the standard shape.
- `req.id` is emitted as `X-Request-ID` in the reply hook before the handler runs.
- Auth middleware (`authenticate`, `requireCapability`, `authenticateRuntime`,
  `authenticatePlatform`) must use the shared helpers — no bare `.send({ error: '...' })`.
- Domain service errors (e.g., `ExtensionNotFoundError`) are mapped to
  `NOT_FOUND` in the controller. Services must not construct HTTP replies.
- Internal throws that are not domain errors produce `INTERNAL` with a generic
  message. The original error is logged at `error` level with `request_id`.

### Helper functions (`apps/api/src/errors/error-reply.ts`)

Each helper accepts `(reply: FastifyReply, message?: string)` and sends the
appropriate HTTP status + body:

| Function                 | Code                | HTTP |
|--------------------------|---------------------|------|
| `sendNotFound`           | `NOT_FOUND`         | 404  |
| `sendInvalidArgument`    | `INVALID_ARGUMENT`  | 400  |
| `sendUnauthenticated`    | `UNAUTHENTICATED`   | 401  |
| `sendPermissionDenied`   | `PERMISSION_DENIED` | 403  |
| `sendConflict`           | `ALREADY_EXISTS`    | 409  |
| `sendResourceExhausted`  | `RESOURCE_EXHAUSTED`| 429  |
| `sendInternal`           | `INTERNAL`          | 500  |
| `sendUnavailable`        | `UNAVAILABLE`       | 503  |

All helpers read `request_id` from the `X-Request-ID` reply header, which is set
before handlers run.

### Client

Clients must read `error` (not `message`) for conditional logic. `message` is
for display only. `request_id` should be forwarded when filing support tickets
or creating automated alerts.

## Structured Log Fields

Every log entry from the API process must include:

| Field        | Type   | Source                          |
|--------------|--------|---------------------------------|
| `time`       | ISO-8601 | Fastify/pino default          |
| `level`      | string | `info`, `warn`, `error`         |
| `msg`        | string | Human-readable message          |
| `req_id`     | string | Fastify `req.id`                |
| `tenant_id`  | string | From JWT or API key claims      |
| `method`     | string | HTTP method (Fastify default)   |
| `url`        | string | Request URL (Fastify default)   |

For error-level logs produced by the global error handler, additionally include:

| Field        | Type   | Description                         |
|--------------|--------|-------------------------------------|
| `err_code`   | string | RPC error code                      |
| `err_name`   | string | Original Error class name           |
| `stack`      | string | Stack trace (never sent to client)  |

The Go freeswitch-agent uses `slog` with equivalent fields: `request_id`,
`tenant_id`, `method`, `url`, `error` (for error values).

## OpenAPI Alignment

The `ErrorResponse` component schema in `docs/api/openapi.yaml` must reflect
this shape. The `scripts/check-openapi-coverage.mjs` script verifies on every
CI run that:

1. Every operation declares a `default` response referencing `ErrorResponse`.
2. The `ErrorResponse` schema has `error`, `message`, and `request_id` fields.
3. All HTTP 4xx/5xx explicit status codes are also covered by `ErrorResponse`.

## Security Constraints

- Stack traces are never included in API responses.
- Internal error details (DB query errors, dependency URLs) are never forwarded.
- `message` must be safe for tenant operators to read without revealing
  infrastructure topology.
